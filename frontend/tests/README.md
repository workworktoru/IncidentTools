# Playwright E2E テスト仕様書

このドキュメントでは、インシデント管理ツールの各モジュール（インシデント・問題・変更・リリース）に実装されているエンドツーエンド（E2E）テストの内容を説明します。

## テストの概要
このテストスイートは、ITIL準拠の各管理プロセスの主要なワークフローが正常に動作することを検証します。

---

## 1. インシデント管理テスト (`tests/incident.spec.ts`)

### 言語切り替えテスト
- **目的**: アプリケーションの国際化（i18n）機能が正しく動作することを確認。
- **検証点**: 言語ボタンクリック後、ヘッダーやメニューが英語/日本語に切り替わること。

### インシデントの作成と編集
- **目的**: インシデントの報告から詳細の更新までの一連の流れを検証。
- **検証点**: 作成後のタイトル表示、編集保存後の成功メッセージ、更新内容の反映。

### インシデントの削除
- **目的**: インシデントを正常に削除できることを確認。
- **検証点**: 削除後のリダイレクト、一覧からの消失。

### ステータス更新
- **目的**: 詳細画面から対応ステータスを更新できることを確認。
- **検証点**: 更新後のステータス値の永続化。

---

## 2. 問題管理テスト (`tests/problem.spec.ts`)

### 問題の作成と更新
- **目的**: 根本原因の特定と回避策の記録プロセスの検証。
- **手順**: 新規問題作成 -> 詳細画面へ遷移 -> 根本原因・回避策の確認 -> ステータスを「Identified」へ更新。
- **検証点**: 入力した内容が詳細画面に正しく表示され、ステータス変更が保存されること。

---

## 3. 変更管理テスト (`tests/change.spec.ts`)

### 変更申請の作成と承認
- **目的**: サービス変更の計画と承認ワークフローの検証。
- **手順**: 新規変更申請作成 -> 影響分析・切り戻し計画の入力 -> 詳細画面へ遷移 -> ステータスを「Approved」へ更新。
- **検証点**: 申請内容の正確な表示、および承認プロセス（ステータス遷移）が正常に行えること。

---

## 4. リリース管理テスト (`tests/release.spec.ts`)

### リリース計画の作成とステータス遷移
- **目的**: ソフトウェアリリースのバージョン管理とデプロイ状況の追跡を検証。
- **手順**: 新規リリース計画作成（バージョン番号・ノート入力） -> 詳細画面へ遷移 -> ステータスを「Building」へ更新。
- **検証点**: バージョン番号の表示形式、リリースノートの保持、ライフサイクル管理の動作。

---

## テストの実行方法

フロントエンドのディレクトリ（`frontend/`）で以下のコマンドを実行します。

```bash
# 全テストを一括実行（Chromium）
npm run test:e2e -- --project=chromium

# 特定のモジュールのみテストする場合
npx playwright test tests/problem.spec.ts --project=chromium

# UIモードでの実行（ブラウザの動作を視覚的に確認）
npx playwright test --ui
```

## 注意事項
- **環境**: `http://localhost:3000` でサーバーが起動している必要があります。
- **データ**: 各テストは独立してデータを生成しますが、実行のたびにDBにテストデータが蓄積されます。定期的なクリーンアップを推奨します。
